# 状态更新模板（自治执行模式）

**使用场景**: 任务完成后自动汇报状态，减少人工确认需求

---

## 模板结构

```markdown
## [TASK-ID] 完成状态更新

### 1. 任务信息
- **任务名称**: 
- **完成时间**: 
- **执行时长**: 
- **资源使用**: 

### 2. 结果摘要（三层结构）

#### Layer 1: 原始数据（客观记录）
```
[可量化指标，不带解释]
例如：
- Configs completed: 300
- r range: [0.001, 1.000]
- Transitions: 15/15
```

#### Layer 2: 分析发现（数据驱动）
```
[从数据中直接观察到的模式]
例如：
- 所有配置均显示 r<0.2 → r>0.8 的跃迁
- 过渡区占比 6%（狭窄）
- K_c 与 σ 正相关
```

#### Layer 3: 判断建议（推论性）
```
[基于Layer 1+2的建议，明确标注不确定性]
例如：
- 建议 E-class 升级为主线候选（待E1 Phase B验证收敛性）
- 若 Phase B 确认滞后效应，可启动 E2/E4
```

### 3. 自动决策建议

| 条件检查 | 状态 | 建议动作 |
|----------|------|----------|
| 结果符合预期 | ☐ | 继续下一任务 |
| 结果超预期 | ☐ | 升级优先级，请求资源调整 |
| 结果不及预期 | ☐ | 进入kill/pivot评估 |
| 发现新 blocker | ☐ | 暂停，请求人工决策 |

### 4. 下一步（已排序）

1. **立即执行**（若Layer 3建议继续）
2. **等待条件**（若依赖其他任务）
3. **人工确认**（若触发gate条件）

### 5. 文件产出

- [ ] 原始数据已落库: `results/xxx/`
- [ ] 分析报告已生成: `docs/xxx.md`
- [ ] TODO已更新: `docs/TODO_MASTER_LIST.md`
- [ ] 状态分层已记录: `docs/STATUS_VERIFICATION.md`

## 使用示例（E1 Phase A风格）

```markdown
## [E1-A] 完成状态更新

### 1. 任务信息
- **任务名称**: E1 Phase A - Critical Coupling Coarse Sweep
- **完成时间**: 2026-03-10 09:13
- **执行时长**: 0.5 min
- **资源使用**: 48-64核并行

### 2. 结果摘要

#### Layer 1: 原始数据
```
Configs: 300 (N=5 × K=20 × σ=3)
Success rate: 300/300 (100%)
r range: [0.001, 1.000]
r<0.2: 148 configs (49.3%)
r>0.8: 134 configs (44.7%)
0.2≤r≤0.8: 18 configs (6.0%)
Transitions: 15/15 configurations
```

#### Layer 2: 分析发现
- 所有(N, σ)组合均显示从低同步到高同步的跃迁
- 过渡区狭窄（6%），暗示锐利相变
- K_c 随 σ 增加而上升：σ=0.1→~0.2, σ=0.5→~1.0, σ=1.0→~1.8

#### Layer 3: 判断建议
- **建议**: E-class升级为主线候选（基于100%相变检测率）
- **条件**: 正式主线地位待E1 Phase B验证K_c随N收敛
- **风险**: 若Phase B显示强滞后效应，需重新评估机制适用性

### 3. 自动决策建议

| 条件检查 | 状态 | 建议动作 |
|----------|------|----------|
| 结果符合预期 | ✅ | 继续 → Phase B |
| 结果超预期 | ☐ | - |
| 结果不及预期 | ☐ | - |
| 发现新 blocker | ☐ | - |

**自动决策**: ✅ 满足继续条件，建议立即启动E1 Phase B

### 4. 下一步

1. **立即执行**: E1 Phase B（临界区加密）
2. **并行准备**: E3 Phase A（Percolation sweep）
3. **排队**: A1×A5（等待E1 Phase B结果）

### 5. 文件产出

- [x] 原始数据: `results/e1_phase_a/sweep_results.csv`
- [x] 分析报告: `docs/candidates/E1_PHASE_A_COMPLETE.md`
- [ ] TODO更新: [待自动更新TODO_MASTER_LIST.md]
- [ ] 状态分层: [待更新STATUS_VERIFICATION.md]
```

## 自治执行触发规则

### 自动继续条件（无需人工确认）

```
IF (前置依赖完成) AND (资源已释放) AND (Layer 3建议继续) AND (无新blocker)
THEN 自动启动下一任务
```

### 必须人工确认的条件

```
IF (结果推翻既有路线) OR (触发kill/pivot/gate) OR (资源互斥) OR (服务器风险)
THEN 暂停，请求人工决策
```

---

## 与人工的沟通协议

### 不需要提问的情况
- ✅ 任务完成，结果符合预期，建议继续
- ✅ 任务完成，结果超预期，建议升级优先级
- ✅ 前置条件满足，按优先级链启动下一任务

### 需要提问的情况
- ⚠️ 任务结果与预期严重不符
- ⚠️ 两个P0任务资源互斥
- ⚠️ 触发kill/continue/archive gate
- ⚠️ 内存/CPU风险（>80%使用）
- ⚠️ 发现新的依赖blocker

## 自治执行规则 v2.0 (2026-03-10)

### 默认执行顺序（E-class STRONG MAINLINE CANDIDATE状态）

**若以下条件满足**：
1. E-class 已标记为 STRONG MAINLINE CANDIDATE
2. E1 Phase B 已完成（滞后效应+双稳态+K_c收敛确认）
3. 无新blocker
4. 资源充足（<80%使用）

**则默认执行顺序为**：

| 顺序 | 任务 | 目标 | 产出 |
|------|------|------|------|
| 1 | **E1 Phase B 深度分析** | hysteresis loop形状, 双稳态区宽度, K_c(N)收敛形式, 临界指数/尺度律 | 判断E-class是"现象级主线"还是"机制级主线" |
| 2 | **001 A1×A5** | 使用D1框架, 钉死write/read gating效果, fixed read伤害是否可被gating消掉 | 001问题层最终定位 |
| 3 | **E3 Revise** | 改模型: 动态网络或K ramp | P→r因果链验证 |
| 4 | **E2/E4 Preparation** | pacemaker emergence, hub knockout after onset | E-class机制深化 |
| 5 | **C1** | 保持后手 | 不抢E-class和001位置 |

**战略排序公式**：
```
E-class > 001 > E3 > E2/E4 > C1
```

**除非以下情况发生，否则不再请求人工确认**：
- 结果推翻既有路线
- 高优先级任务资源互斥
- 触发kill/pivot/archive gate
- 服务器资源风险
- 新证据推翻当前战略排序
