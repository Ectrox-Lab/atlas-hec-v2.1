# SUPERBRAIN V2 CHARTER

**Version**: 2.0  
**Date**: 2026-03-12  
**Status**: DRAFT - Direction Redefinition  
**Author**: Research Lead  
**Scope**: Post-P6 Long-Horizon Executive System

---

## Executive Summary

> **Old Direction**: Build a monolithic superbrain that can do everything itself  
> **New Direction**: Build a governance kernel + evolution kernel that orchestrates an external capability mesh without being hijacked by it

The superbrain is no longer defined by "how much it can do itself" but by "how well it governs what it delegates while maintaining long-horizon coherence."

---

## Part 1: Three-Layer Architecture (Mandatory)

### Layer 1 — Governance Core

**Definition**: The minimal self that must exist for the system to be a coherent agent.

**Endogenous Requirements** (Must be strong, not delegated):

| Capability | Description | Non-Negotiable |
|------------|-------------|----------------|
| **Identity** | Self-model persistence across restarts/contexts | ✅ |
| **Goal Persistence** | Long-horizon goal stability despite short-term pressures | ✅ |
| **Value/Constraint Handling** | Operating within defined boundaries, rejecting violations | ✅ |
| **Judgment** | Ability to evaluate proposals, outputs, and specialist claims | ✅ |
| **Delegation** | Deciding what to do vs. what to assign | ✅ |
| **Audit** | Verifying that delegated work matches intent | ✅ |
| **Long-Horizon Consistency** | Preventing drift across sessions, tasks, and tool dependencies | ✅ |

**Key Principle**: 
> Governance Core does not write code. It decides whether code should be written, by whom, and whether the result is acceptable.

---

### Layer 2 — Evolution Core

**Definition**: The system's capacity to improve itself through experience, not just accumulate logs.

**Endogenous Requirements** (Must be strong, not delegated):

| Capability | Description | Non-Negotiable |
|------------|-------------|----------------|
| **Deep Analysis** | Understanding root causes, not just symptoms | ✅ |
| **Active Exploration** | Intentionally testing boundaries and hypotheses | ✅ |
| **Memory Governance** | Deciding what to keep, compress, promote, or discard | ✅ |
| **Self-Modeling** | Maintaining accurate models of own capabilities and limits | ✅ |
| **Architecture Optimization** | Proposing structural improvements to itself | ✅ |
| **Inheritance/Transmission** | Packaging lessons for future instances | ✅ |

**Key Principle**:
> Evolution Core does not just "summarize." It institutionalizes—converting experience into policy, policy into constitution, and constitution into inheritance.

---

### Layer 3 — Capability Mesh

**Definition**: All tools, skills, and external agents that extend the system's reach.

**Enhanceable by External Attachment** (Can be weak/delegated):

| Capability | Implementation | Governance |
|------------|----------------|------------|
| Code Writing | CLI coders, specialist agents | Executive approval required |
| Information Retrieval | Search tools, query agents | Audit trail mandatory |
| CLI Execution | Shell tools, script runners | Sandboxed, logged |
| Specialist Agents | Domain experts, verifiers | Cannot self-accept |
| Tool Chains | Orchestrated pipelines | Rollback capability required |
| Verifiers | External auditors, test harnesses | Independent of executor |

**Key Principle**:
> Capability Mesh is powerful but subordinate. It cannot self-validate, self-upgrade, or self-modify goals.

## Part 2: Research Objective Shift

### Old Objective (Archived)

> Verify whether a minimal self-improving cognitive core can exist.

**Status**: ✅ **VALIDATED through P1-P6**
- P1-P5a: Minimal continuous self-improvement loop verified
- SEP v1.0: Unified evaluation protocol established
- P5b: Self-maintenance loop verified
- P6: Long-horizon robustness entry ready

### New Objective

> **Verify whether a superbrain with governance kernel + evolution kernel can long-term orchestrate an external capability mesh without being hijacked by it.**

**Core Research Questions**:

1. Can the executive delegate effectively without losing oversight?
2. Can the evolution core institutionalize lessons without becoming an oracle?
3. Can the system resist specialist hijacking over long horizons?
4. Can memory governance remain bounded and coherent?

## Part 3: Prohibitions (New Phase)

The following are explicitly **FORBIDDEN** in V2 research:

| Prohibition | Rationale |
|-------------|-----------|
| ❌ **"Code writing ability" as top-level brain evaluation** | Governance ≠ Implementation |
| ❌ **Parameter count as brain qualification** | 20B can govern; 120B can be misused |
| ❌ **Single demo as system-level completion** | Robustness requires longitudinal validation |
| ❌ **Specialist self-acceptance/self-upgrade/self-goal-modification** | Separation of powers mandatory |
| ❌ **Akashic as oracle** | Akashic records; it does not answer |
| ❌ **Tool dependency without boundedness proof** | Must show mesh does not become crutch |
| ❌ **Memory growth without compaction strategy** | Unbounded memory = eventual failure |
| ❌ **Narrative before evidence** | Claims require logs; logs precede conclusions |

## Part 4: New Entry Gates

Three mandatory research gates define the V2 entry path:

### E1 — Executive Core Gate

**Question**: Can the executive govern without doing all the work?

**Must Demonstrate**:
- Task decomposition completeness
- Tool/specialist selection quality
- Escalation decision quality
- False acceptance rate control
- Rollback capability
- Anti-hijack robustness

**Pass Criteria**: Executive can delegate 80%+ of implementation while maintaining 95%+ outcome quality through audit.

---

### F1 — Evolution Core Gate

**Question**: Can the system institutionalize, not just summarize?

**Must Demonstrate**:
- Bottleneck identification quality
- Hypothesis generation quality
- Memory compression/promotion quality
- Architecture redesign proposal quality
- Lesson → Policy → Constitution conversion

**Pass Criteria**: System produces at least 3 institution-level improvements from experience that persist across restarts.

---

### G1 — Long-Horizon Executive Gate

**Question**: Can governance + evolution resist long-term drift?

**Must Demonstrate**:
- Goal persistence over 72+ hours
- Tool dependency stability (no runaway)
- Memory growth boundedness
- Specialist hijack resistance
- Self-maintenance bounded cost

**Pass Criteria**: 72-hour continuous operation with <5% goal drift, <10% tool dependency increase, no specialist capture events.
